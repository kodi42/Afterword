# Afterword — Master Plan

This is the single source of truth for the build. When working in Claude Code, point it here
first. Keep this file updated as decisions change, so it never drifts from the real app.

---

## 1. What this is

A personal iPhone app for taking reading notes. Two jobs:

1. **Chapter log** — after finishing a chapter, write down what happened and what you thought.
   A running reflection journal, one entry per chapter, per book.
2. **Reference section** — structured notes you flip back to when you forget something mid-read:
   who a character is, what a plot thread is doing, and predictions you want to check later.

Built for reading several books at once (fiction and nonfiction), so the library handles many
in-progress books cleanly.

---

## 2. Decisions locked

| Decision | Choice | Why |
|---|---|---|
| Platform | iPhone via Expo (React Native) | Runs on the phone through Expo Go with no Apple Developer account and no cost. Fast edit-and-reload loop. TypeScript, which fits existing JS experience. |
| Storage | Local only, on device | Personal app. No sync, no server, no auth to build. |
| Database | SQLite via expo-sqlite, queried with Drizzle ORM | Structured, relational, type-safe. Search is a local query, free forever. |
| Notes model | Structured records, not freeform text | Each character/thread/prediction is its own row you can search, sort, and update. Only the name/title is required, so adding an entry stays fast. |
| Cost model | Zero recurring cost | Nothing calls an external API at runtime. AI is a build-time tool only. The finished app runs fully on its own. |
| AI at runtime | None in v1 | Every planned feature is plain local data work. AI stays an optional, on-demand, someday convenience (see Backlog). |

## 3. Data model

Five tables. One book owns everything else. Every child row carries a `bookId` and is deleted
with its book (cascade). Full definitions live in `src/db/schema.ts`.

**books** — the shelf
`id, title, author?, status ('reading'|'finished'), coverUri?, totalChapters?, currentChapter?, startedAt?, finishedAt?, createdAt, updatedAt`

**chapter_notes** — the core reflection log (v1 heart)
`id, bookId, chapterNumber, title?, body, createdAt, updatedAt`

**characters** — reference: who is this again?
`id, bookId, name, description?, firstSeenChapter?, status ('active'|'gone'), createdAt, updatedAt`

**plot_threads** — reference: a plotline tracked across chapters
`id, bookId, title, notes?, status ('open'|'resolved'), createdAt, updatedAt`

**predictions** — reference: a guess you make and later mark right or wrong
`id, bookId, prompt, madeAtChapter?, status ('open'|'answered'), outcome?, wasCorrect?, createdAt, updatedAt`

Relationships: `books 1 --- many chapter_notes / characters / plot_threads / predictions`.

## 4. Navigation and screens

expo-router, where the file path is the route.

- **Library** (`app/index.tsx`) — home. Currently reading up top, finished below. Floating add button.
- **Add book** (`app/book/new.tsx`) — modal. Title required, everything else optional.
- **Book detail** (`app/book/[id].tsx`) — title header, then a segmented control:
  - **Chapters tab** — the reflection log. Add a note for the next chapter, list past notes newest-first.
  - **Reference tab** — sub-sections for Characters, Plot threads, Predictions. (Phase 3.)

Later screens (backlog): note detail/edit, character detail, global search.

## 5. Project structure

Organization rule: **screens stay thin, logic lives in features.** A screen renders and calls a
feature function. It never writes raw SQL.

```
afterword/
├── PLAN.md                     ← this file
├── README.md                   ← setup + run commands
├── app/                        ← screens (route = file path)
│   ├── _layout.tsx             ← runs DB migrations on launch, nav theme
│   ├── index.tsx               ← Library
│   └── book/
│       ├── [id].tsx            ← Book detail (Chapters / Reference tabs)
│       └── new.tsx             ← Add book (modal)
├── src/
│   ├── db/
│   │   ├── schema.ts           ← all tables (the backbone)
│   │   ├── client.ts           ← db handle, cascade pragma
│   │   └── migrations/         ← GENERATED by db:generate, do not hand-edit
│   ├── features/               ← data access + logic, grouped by feature
│   │   ├── books/queries.ts
│   │   ├── chapters/queries.ts
│   │   └── reference/          ← characters / threads / predictions (Phase 3)
│   ├── components/ui/          ← Button, Card, TextField, SegmentedControl, Screen, EmptyState
│   └── theme/                  ← colors, spacing, radius, type tokens
```

**How to add a new feature (the pattern to repeat):**
1. Add or edit the table in `src/db/schema.ts`, then run `npm run db:generate`.
2. Create `src/features/<name>/queries.ts` for its data access.
3. Build the screen or tab in `app/`, using `useLiveQuery` for reactive reads.
4. Reuse components from `src/components/ui`. Add new shared ones there, not inside screens.

## 6. Roadmap

### Phase 0 — Scaffold ✅ (in this repo)
Expo + expo-router + TypeScript + expo-sqlite + Drizzle wired up. Theme, UI components, all
five tables defined, data-access modules for books and chapters. Runs on the phone after the
README setup steps.

### Phase 1 — Library and books
- [x] Library screen: currently reading / finished shelves, empty state, add button
- [x] Add book modal (title, author, total chapters)
- [ ] Edit book (tap-and-hold or an edit button on detail)
- [ ] Mark a book finished; delete a book (with confirm)
- [ ] Show reading progress on each row

### Phase 2 — Chapter log (the core, ship this first) ✅
- [x] Chapters tab: add a note for the next chapter, list notes (newest chapter first)
- [x] Edit and delete a chapter note
- [x] Auto-advance the book's `currentChapter` when a note is added (forward only)
- [x] Long-press a note for edit / delete actions (delete confirms)

**v1 ships at the end of Phase 2.** A working library plus the chapter log is a usable app.

### Phase 3 — Reference section
- [ ] `src/features/reference/` query modules for characters, threads, predictions
- [ ] Characters: list + add/edit, status active/gone, first-seen chapter
- [ ] Plot threads: list + add/edit, open/resolved, filter resolved out
- [ ] Predictions: make a guess, later mark answered + right/wrong, record outcome
- [ ] Reference tab wires these into three sub-sections

### Phase 4 — Search and polish
- [ ] Search across chapter notes and reference entries (start with SQL `LIKE`, graduate to
      SQLite FTS5 for ranked, instant results). Pure local, no AI, no cost.
- [ ] Cover images (expo-image-picker, store the local uri)
- [ ] Finished-books archive view
- [ ] Sort and filter options on the library
- [ ] Export a book's notes to plain text or markdown (share sheet)

## 7. Backlog / someday

Not scheduled. Captured so nothing is lost.

- **On-demand AI helpers (optional, tap-to-run only, never in the background):**
  chapter recap, "everything about this character so far," spoiler-safe summaries. Each would
  be an explicit button, so cost only ever happens when chosen. Some could run on-device on
  newer iPhones with no API at all. Skippable entirely.
- Reading stats (chapters per week, streaks, pace).
- Tags across notes (theme, mood) with tag-based browsing.
- Multiple reference note types beyond the three (settings/places, vocabulary, quotes).
  Quotes were dropped from v1 reference scope but easy to add as a fourth table.
- iCloud or file-based backup/restore, if "just my phone" ever needs to survive a new device.
- Dark theme (second token object, same shape as the light one).
- Home-screen widget showing the current book and last note.

## 8. Conventions

- **Path alias:** import from `@/` (maps to `src/`). e.g. `import { Button } from '@/components/ui'`.
- **Theme only:** no hardcoded colors, spacing, or font sizes in components. Pull from `@/theme`.
- **Reads are reactive:** use `useLiveQuery` so the UI updates itself after a write. No manual refresh.
- **Writes go through features:** screens call `createBook`, `createChapterNote`, etc. Never inline SQL in a screen.
- **Migrations are generated:** change `schema.ts`, run `npm run db:generate`, never edit generated files.
- **One required field per entry:** keep adding things fast; make everything else optional.
